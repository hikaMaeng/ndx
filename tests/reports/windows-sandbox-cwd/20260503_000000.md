## Root Cause

The external tool runner only treated `/...` as an absolute path. On Windows,
host paths such as `C:\Users\hika0\.ndx\system\tools\apply_patch` and
`F:\dev\test1` were not recognized as absolute host paths and could be passed
to Docker as `docker exec -w C:\...`. Linux containers reject that value with
`OCI runtime exec failed: exec failed: Cwd must be an absolute path`.

## Fix Coverage

- Windows workspace paths map to `/workspace`.
- Windows global `.ndx` paths map to `/home/.ndx`.
- Unmapped Windows absolute paths fall back to the sandbox cwd instead of being
  passed to Docker.
- Core tool runtime path normalization handles Windows separators when the tool
  is running inside the sandbox.
- Provider instructions tell models to retry or clearly report file-changing
  tool failures instead of replacing the requested filesystem change with
  manual copy/save instructions.
